# GCU Image Generation — Style Guide

## The Core Idea

Every image in GCU is generated by combining two completely separate things:

1. **What** the image shows — described in an individual `.prompt.txt` file
2. **How** it looks — defined once, globally, in `core/imgen/style_library.py`

This separation means you can regenerate every image in the entire project with an updated visual style by changing a few lines in one file. The individual prompt files never need to change when the style changes.

---

## The Visual Target

Every image in GCU — characters, props, scenes, UI elements — should look like it belongs in a **Pixar or Walt Disney Animation feature film**:

| Reference | Studio | Why it matters |
|-----------|--------|----------------|
| Toy Story | Pixar | Warm character appeal, expressive props |
| The Incredibles | Pixar | Rich cinematic lighting, stylized humans |
| Wall-E | Pixar | Personality in non-human characters |
| Frozen | Walt Disney Animation | Lush environments, soft luminous light |
| Moana | Walt Disney Animation | Vibrant saturated color, depth and warmth |
| Encanto | Walt Disney Animation | Rich textures, expressive faces |

Key qualities that distinguish this look from generic "3D cartoon":

- **Subsurface scattering** — skin, fur, and organic materials glow warmly from within, not just from surface reflection
- **Soft cinematic lighting** — warm key light, gentle fill, no harsh black shadows
- **Appealing rounded forms** — nothing sharp or threatening; even mechanical objects have gentle curves
- **Saturated but harmonious color** — vivid, alive, never garish or neon
- **Full of personality** — a basket, an apple, a puzzle piece should feel like a character would pick it up

What to avoid:

- Photorealism (no product-photography look)
- Flat cell-shading (no 2D cartoon look)
- Generic "cute 3D" (no mobile-game or emoji aesthetic)
- Harsh shadows or dramatic dark contrast
- Busy or cluttered compositions

## File Structure

```
core/imgen/
├── style_library.py      ← THE global style file. Edit this to change how everything looks.
├── generate_asset.py     ← Sends prompt + style to ComfyUI. Don't edit unless fixing a bug.
├── batch_generate.py     ← Runs generate_asset.py on all .prompt.txt in a directory.
├── validate_prompt.py    ← Checks prompt files for required fields and valid style names.
├── remove_bg.py          ← Strips background from rendered images (used for transparent mode).
└── Makefile.template     ← Copy this to any asset directory as "Makefile".

games/puzzle-forest/assets/images/
├── Makefile              ← Copied from Makefile.template. Don't edit — it works as-is.
├── *.prompt.txt          ← One file per image. Describes WHAT the image shows.
└── *.png                 ← Generated output. Delete to force regeneration.

games/robot-lab/assets/images/
├── Makefile              ← Same pattern.
└── *.prompt.txt
```

## The Named Styles

`style_library.py` defines a set of named styles. Each prompt file references one by name. The generator looks it up and appends the full style text to the prompt before sending to ComfyUI.

### `gcu_default`
General purpose. Use this when nothing more specific fits. Covers the Pixar/Disney aesthetic at a broad level. Good starting point for any new asset type.

### `gcu_character`
For characters — animals, people, creatures. Emphasizes expressive eyes, appealing proportions, subsurface scattering on skin and fur, and a warm cinematic key light. Single centered character.

### `gcu_scene`
For full-frame environment images (backgrounds, category illustrations). Emphasizes lush richly-lit settings, storytelling depth, and vibrant saturated color across the whole frame.

### `gcu_prop`
For isolated objects — food, tools, baskets, gadgets. Emphasizes tactile materials with subsurface glow, clean readable silhouette, and the kind of charm that makes an object feel like it belongs in an animated film.

### `gcu_toddler`
For assets aimed at ages 2–4 (Andre, Alanna). Simplified bold shapes, bright primary colors, minimal detail, no shadows. Closer to Disney Junior or Pixar Shorts quality — extremely readable at small sizes.

### `rl_scene`
Robot Lab specific. Full-frame scenes that must contain clearly separated regions of red, green, and blue — required for the color sensor chapter to work correctly (mis-routing channels must produce obviously wrong results).

### `rl_prop`
Robot Lab specific. Electronic components and robot parts in a Pixar/Incredibles/Wall-E style — stylized metallic and plastic materials with teal and gold accents.

## Prompt File Format

Each `.prompt.txt` file has this structure:

```
TARGET FILE: basket-blue.png
SIZE: 1024x1024
BACKGROUND_MODE: transparent
STYLE_NAME: gcu_prop

PROMPT:
A woven basket with blue accents, rounded handle, large open top,
slight three-quarter angle, centered, isolated object.

NOTES:
Color must be unmistakably blue. Single object only, no extra items.
```

### Field reference

| Field | Required | Description |
|-------|----------|-------------|
| `TARGET FILE` | Yes | Output filename (relative to the prompt file's directory) |
| `SIZE` | Yes | Width × height in pixels, e.g. `1024x1024` |
| `BACKGROUND_MODE` | No | `transparent` (removes background after render), `scene` (full-frame, keep background), `solid` (default, plain background) |
| `STYLE_NAME` | Recommended | Name from `style_library.py`. If omitted, uses `gcu_default`. |
| `PROMPT` | Yes | What the image shows. Subject, color, pose, composition only. No style language. |
| `NOTES` | No | Hard constraints the model must not violate. Keep short. |

### What belongs in PROMPT vs STYLE_NAME

**PROMPT describes the subject.** It answers: what is this thing, what color is it, what is it doing, how is it composed?

```
✅ A full-body brown bear facing right, side profile, honey-colored belly,
   standing upright, no scenery, no props, isolated object.

❌ A cute animated bear with soulful eyes, soft painterly fur, warm rim light,
   magical forest glow, appealing silhouette, premium toddler-game quality.
```

The second version mixes style into the subject. Everything after "bear" that describes *how it looks* (not *what it is*) belongs in the style library, not the prompt.

**NOTES add hard constraints.** Short, specific, non-negotiable requirements:

```
✅ Red color must be unmistakable. Must read clearly at 64×64 pixels.
✅ Bear body must be divisible into three horizontal thirds without losing recognizability.
❌ Make it feel warm and inviting with soft magical lighting.  ← that's style, not a constraint
```

## How the Generator Assembles the Final Prompt

`generate_asset.py` builds the text sent to ComfyUI like this:

```
MAIN SUBJECT: <PROMPT block>
| STYLE: <looked-up style text from style_library.py>
| DETAILS: <NOTES block>
| BACKGROUND: <background description>
| CONSTRAINTS: single object, centered composition, no duplicates, no extra objects, no clutter
```

The image model sees all of this as one instruction. The subject tells it *what*, the style tells it *how*.

## Workflow: Generating Images

ComfyUI must be running locally before any generation commands. Start it first:

```bash
source ~/Documents/src/ai/ComfyUI/venv/bin/activate
# then launch ComfyUI in another terminal
```

Then from any asset directory that has a `Makefile`:

```bash
# Validate all prompt files (checks fields and style names)
make validate

# Generate all images that don't exist yet (skips existing output)
make generate

# Generate one specific image
make generate FILE=basket-blue.prompt.txt

# Force regeneration of everything (delete outputs first)
rm *.png && make generate
```

`batch_generate.py` skips any prompt whose output file already exists. Delete the PNG to regenerate it.

## Workflow: Updating the Style

When you want to change how images look — for example, to push the Pixar quality higher, adjust lighting warmth, or change the color palette direction:

1. Open `core/imgen/style_library.py`
2. Edit the relevant style entry (e.g. `gcu_character` to change all animal characters)
3. Delete the existing PNGs you want to regenerate:
   ```bash
   # Regenerate all puzzle-forest images
   rm games/puzzle-forest/assets/images/*.png

   # Or just characters
   rm games/puzzle-forest/assets/images/build-*.png
   rm games/puzzle-forest/assets/images/feed-*.png
   ```
4. Run `make generate` in each asset directory

You do not need to touch any `.prompt.txt` file. The style change propagates automatically because every prompt references the style by name.

## Workflow: Adding a New Asset

1. Create a `.prompt.txt` file in the asset directory:

   ```
   TARGET FILE: my-new-thing.png
   SIZE: 1024x1024
   BACKGROUND_MODE: transparent
   STYLE_NAME: gcu_prop

   PROMPT:
   Describe what the image shows. Colors, pose, composition. Nothing about style.

   NOTES:
   Any hard constraints. Keep short.
   ```

2. Validate it:
   ```bash
   make validate FILE=my-new-thing.prompt.txt
   ```

3. Generate it:
   ```bash
   make generate FILE=my-new-thing.prompt.txt
   ```

The output appears as `my-new-thing.png` (and `my-new-thing-transparent.png` if `BACKGROUND_MODE: transparent`).

## Workflow: Adding a New Game

Each game's asset directory gets its own `Makefile` (copied from `core/imgen/Makefile.template`) and its own `.prompt.txt` files. The style library in `core/imgen/style_library.py` is shared automatically — `PYTHONPATH` in the Makefile points to `core/imgen`.

If a game needs a style that doesn't exist yet, add it to `style_library.py` with a game-specific prefix (e.g. `af_scene` for Art Studio, `sm_card` for SuperMatch3). Use the `gcu_*` styles for everything that fits the general categories.

Steps:
1. Copy `core/imgen/Makefile.template` → `games/my-game/assets/images/Makefile`
2. Create `.prompt.txt` files for each needed image
3. Run `make validate` then `make generate`

## Adding a New Style

Open `core/imgen/style_library.py` and add an entry to the `STYLES` dict:

```python
"my_style_name": (
    "3D animated feature film quality, Pixar and Disney Animation, "
    "... specific style guidance for this asset category ..."
),
```

Then reference it in prompt files with `STYLE_NAME: my_style_name`.

Run `make validate` in any asset directory to confirm the new style name is recognized.

## Background Modes

| Mode | What it does | When to use |
|------|-------------|-------------|
| `transparent` | Renders on white, then `remove_bg.py` strips the background. Output is `filename-transparent.png`. | Characters, props, UI elements that will be composited onto game backgrounds |
| `scene` | Full-frame image, background kept. Output is `filename.png`. | Category illustrations, environment backgrounds |
| `solid` | (default) Plain background kept. | Rarely used in GCU |

For `transparent` mode, the rendered intermediate (white background) is also saved as `filename.png`. The game code should reference `filename-transparent.png`.

## Deterministic Seeds

`generate_asset.py` computes a deterministic seed from the full assembled prompt text. This means:

- The same prompt always produces the same image
- Changing any word in the prompt or style produces a different image
- You can safely delete and regenerate without getting random results

If you want to explore variations of an image without changing the prompt, there is currently no built-in variation mode — the easiest approach is to temporarily add a word to the NOTES field, generate, then remove it.

## Quick Reference

| I want to... | What to do |
|---|---|
| Change the look of all GCU images | Edit `gcu_default` in `core/imgen/style_library.py`, delete PNGs, `make generate` |
| Change how all characters look | Edit `gcu_character`, delete character PNGs, `make generate` |
| Change one image's subject | Edit its `.prompt.txt` PROMPT block, delete its PNG, `make generate` |
| Add a new image | Create `.prompt.txt`, `make validate`, `make generate` |
| Regenerate everything from scratch | `rm *.png && make generate` in the asset directory |
| Check what style a prompt is using | Look for `STYLE_NAME:` in the `.prompt.txt` file |
| Add a new style category | Add entry to `STYLES` dict in `style_library.py` |
| See all available style names | Run `make validate` — any invalid name is reported with the full list |
