# AudioClips — Centralized Narration System for GCU

## Overview

GCU games need spoken narration: puzzle prompts, praise lines, UI cues, completion
fanfares. The current approach uses `window.speechSynthesis` — a synthetic browser
voice that sounds robotic and has inconsistent behavior across iOS versions and
devices.

This document describes the **AudioClips system** — a centralized, game-neutral
module that replaces browser TTS with pre-recorded narration generated by
[piper](https://github.com/rhasspy/piper). The system is designed so that:

1. Every spoken string in every game lives in a stable, human-readable manifest.
2. A single shell script reads those manifests and generates WAV files via piper.
3. The runtime plays WAV files through the existing `AudioManager` (Web Audio API,
   iOS-safe).
4. If a WAV file is missing (e.g., during development before generation), the system
   falls back to `speechSynthesis` so gameplay is never silently broken.

---

## Design Goals

| Goal | Implication |
|------|-------------|
| Centralized, not per-game | One module in `core/audio/` that all games use |
| Voice assignable per game | Puzzle Forest uses `semaine`; future games may use a different voice |
| Single source of truth for strings | The manifest `.txt` files drive both WAV generation and runtime playback |
| No build step | Manifests are plain text; the core module fetches them at runtime |
| Graceful fallback | Missing WAV → speechSynthesis; missing everything → silence |
| Offline-capable | WAV files are bundled in the app; no network requests during gameplay |
| Scriptable generation | A shell script loops over manifest entries and calls piper |

## Manifest Format

Already in use. One file per thematic group, one entry per line:

```
key|Spoken text here.
```

**Rules:**
- Keys are lowercase, underscores only, stable (never renamed after WAVs are
  generated, because the filename IS the key).
- Text is exactly what piper will speak — punctuation, capitalization, and rhythm
  matter. Edit the text to tune delivery; then regenerate.
- Blank lines and lines starting with `#` are ignored (comments allowed).
- No quotes around text; the pipe `|` is the only delimiter.

**Example (`gameplay-prompts.txt`):**
```
# Build the Animal prompts
build_bear|Can you build the bear?
build_fox|Can you build the fox?
build_owl|Can you build the owl?
build_rabbit|Can you build the rabbit?
build_duck|Can you build the duck?
build_squirrel|Can you build the squirrel?
```

## Per-Game Configuration: `audio-config.json`

Each game that uses AudioClips has one config file at
`games/{game}/assets/audio/audio-config.json`. This file serves two purposes:

1. Tells the **generation script** which manifests to process and which piper voice
   to use.
2. Tells the **runtime module** which manifest files to load and where the WAV clips
   directory is.

```json
{
  "gameId": "puzzle-forest",
  "voice": "semaine",
  "voiceModel": "en_GB-semaine-medium.onnx",
  "voicesDir": "~/piper/voices/semaine",
  "manifests": [
    "gameplay-prompts.txt",
    "praise-lines.txt",
    "build-complete-lines.txt",
    "mission-complete-lines.txt",
    "retry-lines.txt",
    "ui-lines.txt"
  ],
  "clipsDir": "clips"
}
```

**Fields:**

| Field | Used by | Purpose |
|-------|---------|---------|
| `gameId` | both | Namespaces clips so multiple games can coexist |
| `voice` | script, docs | Human-readable voice name |
| `voiceModel` | script | Filename of the `.onnx` model inside `voicesDir` |
| `voicesDir` | script | Local machine path to the piper voice directory |
| `manifests` | both | Ordered list of manifest filenames to load/process |
| `clipsDir` | both | Subdirectory (relative to `assets/audio/`) for WAV output |

> **Note:** `voicesDir` is a local laptop path. It is only read by the generation
> script, never by the browser.

## Core Module: `AudioClips.js`

Lives at `core/audio/AudioClips.js`. Singleton, like `AudioManager`.

### Responsibilities

- Load a game's `audio-config.json` and all its manifest files via `fetch`.
- Build an in-memory map: `"puzzle-forest:feed_bunny"` → `"Can you feed the bunny?"`.
- When asked to speak a key:
  1. Attempt to load and play the WAV file from the `clips/` directory.
  2. If the WAV is absent or fails to load, fall back to `speechSynthesis`.
  3. If `speechSynthesis` is also unavailable, do nothing (silent graceful failure).
- Cancel any in-progress speech when the scene exits.

### API

```javascript
AudioClips.getInstance()
```
Returns the singleton. (Same pattern as `AudioManager`.)

---

```javascript
await AudioClips.loadGame(gameId, configBasePath)
```
Fetches `audio-config.json` and all listed manifest files for the game.
Call this once during the game's initialization (before any narration is needed).
Safe to call again — duplicate calls for the same `gameId` are ignored.

- `gameId`: string, e.g. `'puzzle-forest'`
- `configBasePath`: URL path to the `assets/audio/` directory,
  e.g. `'games/puzzle-forest/assets/audio'`

---

```javascript
AudioClips.speak(gameId, key, { rate, pitch, volume } = {})
```
Plays the clip identified by `key` under `gameId`. The options (`rate`, `pitch`,
`volume`) are only used by the speechSynthesis fallback path — piper WAVs are fixed
recordings. Returns immediately; playback is fire-and-forget.

---

```javascript
AudioClips.speakRandom(gameId, prefix, opts = {})
```
Picks a random registered key whose name starts with `prefix` and speaks it.
Used for praise lines (`'praise_'`), retry lines (`'retry_'`), etc.

---

```javascript
AudioClips.cancel()
```
Stops any in-progress WAV playback or speechSynthesis utterance. Call from
every `scene.exit()`.

---

```javascript
AudioClips.isAvailable(gameId, key)
```
Returns `true` if a WAV file for this key has been successfully loaded into
`AudioManager`. Useful for debugging or showing a visual indicator that audio is
live vs. synthesized.

---

```javascript
AudioClips.exportManifest(gameId)
```
Returns a plain-text string of all registered `key|text` pairs for the game.
Useful during development to audit what's registered (`console.log` it).

---

### Internal Flow: `speak(gameId, key)`

```
speak('puzzle-forest', 'build_fox')
  │
  ├─ Look up text: "Can you build the fox?"
  │
  ├─ AudioContext ready?
  │    NO  → TTS fallback (don't cache failure — will retry on next call)
  │
  ├─ Construct WAV path:
  │    games/puzzle-forest/assets/audio/clips/build_fox.wav
  │
  ├─ WAV already loaded in AudioManager?
  │    YES → AudioManager.play('clips:puzzle-forest:build_fox')  ✓ done
  │    NO  → audio.load(key, wavUrl)
  │            SUCCESS (sounds.has(key)) → play()  ✓ done
  │            FAIL (not in sounds)     → mark failed, fall through
  │
  └─ TTS fallback:
       new SpeechSynthesisUtterance(text) → speechSynthesis.speak(...)
       (or silence if speechSynthesis is absent)
```

## Generation Pipeline

### The Script: `scripts/generate-audio.sh`

A shell script that takes a game ID, finds its `audio-config.json`, iterates over
all manifest entries, and calls piper once per line.

**Usage:**

```bash
# Generate all clips for Puzzle Forest
./scripts/generate-audio.sh puzzle-forest

# Generate clips for all games
./scripts/generate-audio.sh --all

# Dry run: show what would be generated without running piper
./scripts/generate-audio.sh puzzle-forest --dry-run

# Force regeneration even if WAV already exists
./scripts/generate-audio.sh puzzle-forest --force
```

**What it does, step by step:**

1. Locate piper (checks `$PIPER` env var, then common install locations).
2. Read `games/{gameId}/assets/audio/audio-config.json`.
3. Create `games/{gameId}/assets/audio/clips/` if it doesn't exist.
4. For each manifest file listed in `config.manifests`:
   - Read the file line by line.
   - Skip blank lines and `#` comments.
   - Split each line on the first `|` to get `key` and `text`.
   - If `clips/{key}.wav` already exists and `--force` is not set, skip it.
   - Run piper:
     ```bash
     echo "text" | piper \
       --model {voicesDir}/{voiceModel} \
       --output_file games/{gameId}/assets/audio/clips/{key}.wav
     ```
5. Print a summary: N generated, M skipped.

**Why skip existing files by default?**
Because piper takes a few seconds per clip. `--force` is available when you've
edited text and need to re-record specific lines.

## How a Game Uses AudioClips

### Step 1: Load at game init

In the game's `index.js`, fire-and-forget during `initGame()` — the manifests
(tiny text files) will finish loading long before the user reaches the first puzzle:

```javascript
import { AudioClips } from '../../core/audio/index.js';

export async function initPuzzleForest(sceneManager) {
  // ... register scenes ...
  AudioClips.getInstance().loadGame('puzzle-forest', 'games/puzzle-forest/assets/audio');
}
```

### Step 2: Speak a key

Replace any direct `speechSynthesis.speak(...)` call with:

```javascript
AudioClips.getInstance().speak('puzzle-forest', 'build_fox');
```

For praise / retry lines, use the random helper:

```javascript
AudioClips.getInstance().speakRandom('puzzle-forest', 'praise_');
```

### Step 3: Cancel on exit

In every scene's `exit()`, replace `window.speechSynthesis.cancel()` with:

```javascript
AudioClips.getInstance().cancel();
```

## Clip Key Convention: Derived from Puzzle ID

For game prompts, the clip key is derived directly from the puzzle's data ID by
replacing hyphens with underscores:

```
puzzle.id  →  clip key
feed-bunny →  feed_bunny
build-fox  →  build_fox
sort-red   →  sort_red
```

This means the manifest and the game data stay in sync automatically — no separate
mapping table needed.

## Fallback Behavior

| WAV exists? | speechSynthesis available? | Result |
|-------------|---------------------------|--------|
| Yes | — | WAV plays via Web Audio (iOS-safe, consistent) |
| No | Yes | Browser TTS speaks the registered text |
| No | No | Silent — no crash, no error |

**During development** (before running the generation script): all clips fall back to
speechSynthesis, so the game is fully playable.

**In production** (after running the script): all clips play from pre-recorded WAV
files. The fallback path is never hit.

## Adding a New Game

- [ ] Create `games/{game}/assets/audio/` directory
- [ ] Create `audio-config.json` with the chosen voice and manifest list
- [ ] Create manifest `.txt` files with all spoken strings
- [ ] Create `games/{game}/assets/audio/clips/` directory (empty initially)
- [ ] In `games/{game}/index.js`, call `AudioClips.getInstance().loadGame(...)`
- [ ] Replace all `speechSynthesis` calls with `AudioClips.speak(gameId, key)`
- [ ] Replace all `speechSynthesis.cancel()` calls with `AudioClips.cancel()`
- [ ] Run `./scripts/generate-audio.sh {gameId}` to generate WAV files
- [ ] Test on Safari/iOS — confirm WAVs play, confirm fallback works if a file is removed

## Future Considerations

- **Preloading**: Call `audio.load()` for all clips at scene enter to avoid first-play
  fetch latency. Acceptable trade-off once clip count is known.
- **Subtitles**: The text for every spoken line is already registered — displaying
  on-screen captions in sync with audio is a natural extension.
- **Volume normalization**: piper output levels may vary. A normalization pass using
  `ffmpeg -af loudnorm` in the generation script would make all clips perceptually
  equal.
- **Bundle size**: Each WAV is ~20–80 KB at piper's default 22050 Hz. 100 clips ≈
  2–8 MB — well within the 50 MB app budget.
